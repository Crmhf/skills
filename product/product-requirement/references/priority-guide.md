# 优先级划分指南

详细说明如何科学地为产品需求分配优先级。

## 优先级框架

### RICE 评分模型

RICE = Reach × Impact × Confidence / Effort

| 维度 | 说明 | 评分标准 |
|------|------|----------|
| **Reach** (触达) | 功能影响的用户数量 | 每季度/每月影响用户数 |
| **Impact** (影响) | 对用户的价值程度 | 3=巨大, 2=高, 1=中, 0.5=低, 0.25=极低 |
| **Confidence** (信心) | 对预估的确信程度 | 100%=高, 80%=中, 50%=低 |
| **Effort** (成本) | 实现所需资源 | 人月/人周 |

**计算示例：**

| 功能 | Reach | Impact | Confidence | Effort | RICE得分 |
|------|-------|--------|------------|--------|----------|
| 微信登录 | 5000用户 | 2(高) | 80% | 1人周 | 8000 |
| 数据统计 | 100用户 | 3(巨大) | 50% | 2人月 | 75 |
| 暗黑模式 | 2000用户 | 1(中) | 100% | 2人周 | 1000 |

### MoSCoW 方法

| 优先级 | 含义 | 处理策略 |
|--------|------|----------|
| **M**ust have | 必须有 | 没有它产品无法发布 |
| **S**hould have | 应该有 | 很重要但可妥协 |
| **C**ould have | 可以有 | 有更好，没有也行 |
| **W**on't have | 不会有 | 明确排除，未来考虑 |

映射到P级别：
- Must → P0
- Should → P1
- Could → P2/P3
- Won't → 暂不实现

## 优先级决策矩阵

### 价值 vs 成本矩阵

```
            高价值
              │
    快速获胜   │   重点投入
    (P1优先)   │   (P0核心)
              │
低 effort ────┼────────── 高 effort
              │
    低优先级   │   需要论证
    (P3)       │   (P2/重评估)
              │
            低价值
```

### 用户价值 vs 业务价值

| 用户价值＼业务价值 | 低 | 中 | 高 |
|-------------------|----|----|----|
| **高** | P2 | P1 | P0 |
| **中** | P3 | P2 | P1 |
| **低** | 不做 | P3 | P2 |

## 场景化优先级判断

### P0 - 核心功能（Must Have）

**判断标准：**
- 没有它用户无法完成核心任务
- 产品发布的最小可用集(MVP)
- 竞品都有的基础功能

**典型场景：**
- 电商：浏览商品、下单支付
- 社交：注册登录、添加好友、发送消息
- SaaS：创建项目、数据保存、基础协作

**比例控制：** 占全部需求的 15-20%

### P1 - 重要功能（Should Have）

**判断标准：**
- 显著提升用户体验
- 影响用户留存和转化
- 相比竞品的基础差异化

**典型场景：**
- 电商：优惠券、订单跟踪、评价系统
- 社交：朋友圈、语音消息、表情包
- SaaS：权限管理、数据导出、邮件通知

**比例控制：** 占全部需求的 30-40%

### P2 - 增强功能（Could Have）

**判断标准：**
- 锦上添花的功能
- 特定用户群体需要
- 可以延后实现

**典型场景：**
- 电商：积分系统、会员等级、直播带货
- 社交：语音房、短视频、小程序
- SaaS：高级报表、自动化工作流、API集成

**比例控制：** 占全部需求的 30-35%

### P3 - 扩展功能（Won't Have Now）

**判断标准：**
- 长期规划中的功能
- 边缘场景支持
- 创新探索性需求

**典型场景：**
- 电商：AR试衣、区块链溯源、元宇宙店铺
- 社交：VR社交、AI虚拟人、脑机接口
- SaaS：AI智能助手、预测分析、区块链存证

**比例控制：** 占全部需求的 10-15%

## 常见优先级误区

### ❌ 误区1：老板说的都是P0

**问题：** 所有需求都被标记为最高优先级，失去优先级意义。

**解决：** 用数据说话，展示资源限制，建议分批实现。

### ❌ 误区2：先做简单的

**问题：** 为了快速交付先做简单功能，核心功能被延误。

**解决：** 优先保障P0功能，简单功能可以并行开发。

### ❌ 误区3：技术驱动优先级

**问题：** 因为技术有趣就做，而非用户需要。

**解决：** 坚持用户价值和业务价值导向。

### ❌ 误区4：忽视技术债务

**问题：** 只关注新功能，不安排重构和优化。

**解决：** 将技术优化作为独立需求纳入排期。

## 优先级沟通技巧

### 与开发团队沟通

1. **解释业务价值**：为什么这个功能重要
2. **说明用户场景**：具体什么用户会用
3. **提供数据支撑**：预期使用量、转化率影响
4. **接受技术评估**：开发可能对成本有不同判断

### 与业务方沟通

1. **资源可视化**：展示团队容量和已有排期
2. **替代方案**：高成本需求的低成本替代
3. **分期实现**：将大需求拆分为多期
4. **数据验证**：建议先做MVP验证价值

### 优先级调整模板

```
需求：【功能名称】

原优先级：P1
建议优先级：P2

调整原因：
1. 【业务原因】如：当前季度重点是拉新，此功能是留存向
2. 【资源原因】如：需要2人月，但Q3只剩1人月可用
3. 【依赖原因】如：依赖的第三方API还未就绪

建议方案：
- 方案A：简化功能，保持P1，实现核心流程
- 方案B：推迟到Q4，配合其他留存功能一起上线
- 方案C：使用替代方案【XXX】，成本降低50%
```

## 优先级动态管理

### 定期Review机制

| Review类型 | 频率 | 参与者 | 重点 |
|------------|------|--------|------|
| 站会同步 | 每日 | 团队 | 当日优先级确认 |
| 迭代规划 | 每双周 | 产品+开发 | 迭代内需求调整 |
| 季度规划 | 每季度 | 全员 | 大方向优先级调整 |
| 临时评审 | 按需 | 相关方 | 紧急插入评估 |

### 优先级变更记录

| 日期 | 需求ID | 原优先级 | 新优先级 | 变更原因 | 决策人 |
|------|--------|----------|----------|----------|--------|
| 2024-01-15 | F-123 | P1 | P0 | 竞品已上线类似功能，市场压力大 | 张三 |
| 2024-01-20 | F-456 | P2 | 暂停 | 数据反馈不佳，需重新调研 | 李四 |

### 插入需求处理流程

```
新需求提出
    │
    ▼
紧急评估：是否影响当前迭代目标？
    │
    ├── 是 → 与PO/开发leader协商，替换低优先级需求
    │
    └── 否 → 放入需求池，下次迭代评估
```

## 实战检查清单

- [ ] P0需求是否覆盖了产品核心使用流程
- [ ] P0需求占比是否控制在20%以内
- [ ] 每个优先级都有清晰的划分标准
- [ ] 需求之间依赖关系是否梳理清楚
- [ ] 是否考虑了技术实现成本
- [ ] 业务方是否认可当前优先级
- [ ] 是否有处理紧急插入的机制
