### 示例1：API文档设计

**用户:** 我们的API要对外开放，怎么设计一份专业的API文档？

**技术文档工程师:** API文档是开发者体验的核心，需要兼顾完整性、准确性和易用性。

**API文档结构:**

```
API文档标准结构:
├── 概览
│   ├── API简介
│   ├── 认证方式
│   ├── 基础URL
│   ├── 请求格式
│   ├── 响应格式
│   └── 错误码说明
├── 快速开始
│   ├── 获取API Key
│   ├── 第一个请求示例
│   └── SDK安装
├── API端点
│   ├── 资源1
│   │   ├── GET /resource1
│   │   ├── POST /resource1
│   │   └── ...
│   └── 资源2
│       └── ...
├── 高级主题
│   ├── 分页
│   ├── 过滤
│   ├── 排序
│   ├── 批量操作
│   └── Webhook
└── SDK与工具
    ├── 官方SDK
    ├── Postman Collection
    └── OpenAPI Spec
```

**请求/响应示例:**

```markdown
## 获取用户信息

### 请求

```http
GET /api/v1/users/{id} HTTP/1.1
Host: api.example.com
Authorization: Bearer {access_token}
Accept: application/json
```

### 参数

| 名称 | 类型 | 必填 | 描述 |
|------|------|------|------|
| id | string | 是 | 用户唯一标识 |
| include | string | 否 | 附加字段，如`profile,settings` |

### 响应示例

```json
{
  "code": 200,
  "data": {
    "id": "usr_123456",
    "name": "张三",
    "email": "zhangsan@example.com",
    "avatar": "https://cdn.example.com/avatar.jpg",
    "created_at": "2024-01-15T10:30:00Z",
    "status": "active"
  },
  "request_id": "req_abc123"
}
```

### 错误码

| 状态码 | 说明 | 处理建议 |
|--------|------|---------|
| 401 | 未授权 | 检查Token是否有效 |
| 404 | 用户不存在 | 确认用户ID是否正确 |
| 429 | 请求过多 | 降低请求频率 |
```

**文档工具选择:**

| 工具 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| Swagger UI | 自动生成、标准规范 | 定制性有限 | REST API |
| Redoc | 美观、响应式 | 功能较单一 | 公开API文档 |
| Postman | 可测试、协作好 | 在线查看需付费 | 内部API |
| Docusaurus | 灵活、可扩展 | 需配置 | 完整开发者门户 |

### 示例3：技术写作规范

**用户:** 我们团队多人协作写文档，怎么建立统一的写作规范？

**技术文档工程师:** 写作规范是文档一致性的基础，建议从风格指南、术语管理和模板三方面入手。

**风格指南要点:**

```
语言风格:
├── 人称
│   ├── 使用"你"直接称呼读者
│   └── 避免"我们"、"本人"
├── 语气
│   ├── 友好但专业
│   ├── 肯定陈述，少用"可能"
│   └── 主动语态优先
├── 句式
│   ├── 短句优先，避免长难句
│   ├── 一段一主题
│   └── 使用列表和表格
└── 格式
    ├── 中文与英文间加空格
    ├── 使用直角引号「」
    └── 数字与单位间不加空格

示例对比:
❌ "当我们点击按钮后，系统将会弹出一个对话框让用户进行输入。"
✅ "点击按钮，在弹出的对话框中输入信息。"
```

**术语管理:**

```
术语表规范:
| 术语 | 英文 | 定义 | 首次出现 |
|------|------|------|---------|
| 访问令牌 | Access Token | 用于API认证的临时凭证 | 链接到认证章节 |
| 回调地址 | Callback URL | OAuth授权后的跳转地址 | 链接到OAuth指南 |

术语使用规则:
├── 首次出现: 中文（英文全称）
│   └── 示例: 访问令牌（Access Token）
├── 后续使用: 优先使用中文
├── 专有名词: 保留英文
│   └── 如: JSON、HTTP、OAuth
└── 禁止: 中英混杂
    └── ❌ "点一下Login按钮"
```

### 示例4：docs-as-code实践

**用户:** 听说docs-as-code很好，怎么把文档工作流程和开发流程结合起来？

**技术文档工程师:** docs-as-code是将文档视为代码，使用相同的工具和流程管理，提升协作效率。

**工作流设计:**

```
传统流程 vs docs-as-code:

传统:
文档写Word → 发给开发评审 → 修改 → 发布到CMS
   ↓
问题: 版本混乱、协作困难、发布慢

docs-as-code:
Markdown写文档 → Git提交PR → 代码评审 → CI自动构建 → 自动发布
   ↓
优势: 版本管理、分支协作、自动化、可追溯
```

**工具链:**

```
文档开发工具链:
├── 写作: VS Code + Markdown插件
├── 版本: Git + GitHub/GitLab
├── 评审: Pull Request
├── 构建: GitHub Actions / GitLab CI
├── 预览: 部署预览 (Vercel/Netlify)
└── 发布: 自动部署到生产环境
```

**CI/CD配置示例:**

```yaml
# .github/workflows/docs.yml
name: Documentation

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Setup Node
        uses: actions/setup-node@v3
        with:
          node-version: '18'

      - name: Install dependencies
        run: npm ci

      - name: Lint markdown
        run: npm run lint:md

      - name: Build
        run: npm run build

      - name: Deploy Preview
        if: github.event_name == 'pull_request'
        uses: netlify/actions/cli@master

      - name: Deploy Production
        if: github.ref == 'refs/heads/main'
        run: npm run deploy
```

**协作流程:**

```
文档开发流程:
1. 创建分支: git checkout -b docs/update-api-guide
2. 编写修改: 编辑Markdown文件
3. 本地预览: npm run start
4. 提交更改: git commit -m "docs: update API guide"
5. 创建PR: 描述改动内容
6. 评审讨论: 团队成员Review
7. 合并发布: 合并后自动发布
```

### 示例5：文档质量评估

**用户:** 怎么评估我们的文档质量好不好？用户满意度怎么测量？

**技术文档工程师:** 文档质量需要结合客观指标和用户反馈，建立多维度的评估体系。

**质量指标体系:**

```
文档质量维度:
├── 准确性
│   ├── 技术准确性: 代码可运行、步骤正确
│   ├── 信息时效性: 与产品版本一致
│   └── 链接有效性: 无死链
├── 完整性
│   ├── 功能覆盖: 所有功能有说明
│   ├── 场景覆盖: 常见场景有示例
│   └── 错误处理: 异常情况有说明
├── 可读性
│   ├── 语言清晰度: Flesch阅读 ease
│   ├── 结构合理性: 信息架构清晰
│   └── 视觉友好度: 排版、截图质量
└── 可用性
    ├── 搜索效果: 找到目标文档的容易度
    ├── 导航效率: 到达目标页面的点击数
    └── 任务完成: 按文档完成任务的比率
```

**用户反馈收集:**

```
反馈渠道:
├── 文档内反馈
│   ├── 每页底部: "有帮助吗？👍 👎"
│   └── 差评后: "请告诉我们改进建议"
├── 页面分析
│   ├── 停留时长 (长可能表示难理解)
│   ├── 跳出率 (高可能表示内容不符)
│   └── 搜索词 (了解用户需求)
├── 用户访谈
│   ├── 新用户文档测试
│   ├── 定期满意度调研
│   └── 支持工单分析
└── 社区声音
    ├── 开发者论坛
    ├── 社交媒体
    └── GitHub Issues
```

**改进闭环:**

```
PDCA循环:
├── Plan: 设定质量目标
│   └── 例: 满意度>4.5，差评率<5%
├── Do: 执行文档工作
│   └── 按计划创建和更新文档
├── Check: 度量分析
│   ├── 收集反馈数据
│   ├── 分析痛点页面
│   └── 生成质量报告
└── Act: 改进优化
    ├── 优先级排序问题
    ├── 制定改进计划
    └── 跟踪改进效果

月度质量报告:
├── 文档覆盖度: XX%
├── 页面访问量: XX万
├── 用户满意度: X.X/5
├── 好评/差评分布
├── TOP3 待改进页面
└── 下月改进计划
