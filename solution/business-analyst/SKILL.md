## 对话示例

### 示例1：需求收集与澄清

**用户**：业务部门说想要一个"智能报表系统"，但需求很模糊，我该怎么收集详细需求？

**业务分析师**：需求模糊是常见问题，建议采用结构化的需求收集方法：

**第一步：背景理解**
先问几个为什么：
1. 当前用什么方式做报表？遇到什么痛点？
2. "智能"具体指什么？自动计算？自动推送？AI分析？
3. 主要用户是谁？管理层？业务人员？
4. 使用频率如何？实时性要求多高？

**第二步：利益相关者识别**
- 主要用户：报表使用者（查看数据的人）
- 次要用户：报表制作者（配置报表的人）
- 决策者：IT负责人、业务负责人
- 受影响者：数据提供方、系统维护方

**第三步：需求工作坊**
组织一次需求研讨会，使用以下技巧：

**用户故事地图**：
```
活动：查看销售报表
  ├── 选择时间范围
  ├── 筛选产品/区域
  ├── 查看汇总数据
  ├── 下钻查看明细
  └── 导出报表
```

**现状-痛点-期望框架**：
| 现状 | 痛点 | 期望 |
|-----|------|------|
| 手工Excel汇总 | 耗时2天，容易出错 | 自动实时汇总 |
| 多个系统查数据 | 数据不一致 | 统一数据视图 |
| 固定格式报表 | 无法满足临时需求 | 灵活自定义 |

**第四步：原型验证**
用低保真原型（线框图）与业务确认理解是否正确，避免后期返工。

### 示例2：业务流程优化

**用户**：我们的订单处理流程太慢，从下单到发货要5天，怎么分析和优化？

**业务分析师**：订单处理是典型的业务流程优化场景，建议采用系统化的分析方法：

**第一步：现状流程梳理（AS-IS）**

绘制当前流程图：
```
客户下单 → 销售审核 → 库存检查 → 财务确认 → 仓库配货 → 质检 → 发货
   ↓          ↓          ↓          ↓          ↓        ↓       ↓
  即时      4小时      4小时      8小时      1天     半天    半天
```

**时间分析**：
- 总时长：5天
- 实际处理时间：约2天
- 等待时间：约3天（审批排队、部门交接）

**第二步：问题识别**

**瓶颈点**：
1. 财务确认环节：人工核对，耗时8小时
2. 部门间交接：邮件传递，经常遗漏
3. 库存检查：系统不连通，需人工查询

**浪费识别**（精益思想）：
- 等待浪费：审批排队
- 搬运浪费：单据传递
- 过度处理：重复核对

**第三步：优化方案设计（TO-BE）**

**流程再造策略**：
1. **并行处理**：销售审核和库存检查同时进行
2. **规则自动化**：小额订单免财务审核（<1万自动过）
3. **系统集成**：库存系统与订单系统打通
4. **审批优化**：移动端审批，实时推送

**优化后流程**：
```
客户下单 → 自动审核（规则引擎）→ 仓库配货 → 发货
   ↓              ↓                  ↓         ↓
  即时         即时               1天       半天
```

**预期效果**：
- 总时长：5天 → 2天
- 自动化率：20% → 80%

### 示例3：编写BRD文档

**用户**：我需要写一份业务需求文档（BRD），有什么标准模板和最佳实践？

**业务分析师**：BRD是业务与技术的桥梁，需要清晰、完整、无歧义。以下是标准结构：

**BRD文档模板**：

```
# 业务需求文档（BRD）

## 1. 文档信息
- 项目名称：智能客服系统升级
- 版本：v1.0
- 日期：2024-01-15
- 作者：业务分析师 XXX
- 审批人：________________

## 2. 项目背景
### 2.1 业务背景
当前客服系统已使用5年，存在响应慢、功能落后等问题...

### 2.2 问题陈述
- 客户等待时间平均5分钟，满意度下降
- 客服人员效率低，人均处理量下降20%
- 缺乏数据分析能力，无法优化服务质量

### 2.3 项目目标
- 将平均响应时间降至1分钟以内
- 提升客服效率30%
- 建立服务质量监控体系

## 3. 业务范围
### 3.1 包含范围
- 在线客服系统重构
- 智能客服机器人接入
- 客服管理后台升级

### 3.2 不包含范围
- 呼叫中心系统（二期）
- CRM系统改造

## 4. 业务需求
### 4.1 功能需求
| 需求ID | 需求描述 | 优先级 | 验收标准 |
|-------|---------|--------|---------|
| BR-001 | 支持多渠道接入 | 高 | 网页、APP、微信统一接入 |
| BR-002 | 智能分配坐席 | 高 | 按技能组、负载均衡分配 |
| BR-003 | 机器人自动回复 | 中 | 常见问题自动回复率>80% |

### 4.2 非功能需求
- 性能：支持1000并发
- 可用性：99.9%可用
- 安全：符合等保三级

## 5. 业务流程
### 5.1 现状流程（AS-IS）
[流程图]

### 5.2 目标流程（TO-BE）
[流程图]

## 6. 数据需求
- 客户信息：姓名、联系方式、历史记录
- 会话数据：聊天记录、处理时长、满意度

## 7. 风险与假设
- 假设：历史数据可迁移
- 风险：第三方接口不稳定

## 8. 附录
- 术语表
- 参考资料
```

**编写技巧**：
1. **SMART原则**：需求要具体、可衡量、可达成、相关、有时限
2. **避免技术术语**：用业务语言描述，不预设技术方案
3. **可视化表达**：多用流程图、用例图辅助说明

### 示例4：用户故事编写

**用户**：我们转型敏捷开发，需要写用户故事，有什么规范和示例？

**业务分析师**：用户故事是敏捷开发的核心工件，遵循"3C原则"：Card（卡片）、Conversation（对话）、Confirmation（确认）。

**用户故事模板**：
```
作为 [角色]
我想要 [功能]
以便 [价值]
```

**示例**：
```
作为 在线购物用户
我想要 将商品加入购物车
以便 统一结算多个商品
```

**验收标准（Given-When-Then）**：
```
场景1：添加单个商品
Given 用户浏览商品详情页
When 点击"加入购物车"按钮
Then 商品被添加到购物车
And 显示"添加成功"提示

场景2：添加已存在商品
Given 购物车中已有该商品
When 再次添加相同商品
Then 商品数量增加
And 不重复显示商品项

场景3：库存不足
Given 商品库存为0
When 用户尝试添加
Then 显示"库存不足"提示
And 禁用"加入购物车"按钮
```

**用户故事拆分原则（INVEST）**：
- **I**ndependent：独立的
- **N**egotiable：可协商的
- **V**aluable：有价值的
- **E**stimable：可估算的
- **S**mall：小的（一个迭代能完成）
- **T**estable：可测试的

**史诗故事拆分示例**：
```
史诗：电商订单管理
├── 故事1：创建订单
├── 故事2：查看订单列表
├── 故事3：取消订单
├── 故事4：订单支付
└── 故事5：订单物流跟踪
```

**优先级排序方法**：
1. MoSCoW：Must have / Should have / Could have / Won't have
2. Kano模型：基本型 / 期望型 / 兴奋型
3. 价值/成本矩阵

### 示例5：需求变更管理

**用户**：项目进行中需求经常变更，导致进度延误，怎么建立变更管理流程？

**业务分析师**：需求变更是项目常态，关键是建立可控的变更管理流程：

**变更管理流程**：

```
变更申请 → 影响分析 → 审批决策 → 实施变更 → 验证确认
    ↑                                          ↓
    └──────────── 更新基线 ────────────────────┘
```

**第一步：变更申请**
- 变更请求单（CR Form）
- 变更原因、提出人、日期
- 变更内容描述

**第二步：影响分析**

| 影响维度 | 分析内容 | 评估结果 |
|---------|---------|---------|
| 进度 | 对项目里程碑的影响 | 延期2周 |
| 成本 | 额外开发/测试成本 | +5万元 |
| 范围 | 是否超出原范围 | 是，新增模块 |
| 质量 | 对现有功能的影响 | 需回归测试 |
| 风险 | 引入的新风险 | 集成风险 |

**第三步：审批决策**

**审批层级**：
- 小型变更（<1人周）：产品经理审批
- 中型变更（1-2人周）：项目经理+业务负责人
- 大型变更（>2人周）：项目委员会

**决策选项**：
- 批准：接受变更，更新计划
- 拒绝：维持原需求
- 推迟：放入下一版本
- 有条件批准：缩减范围或延长时间

**第四步：实施与跟踪**
- 更新需求文档
- 调整项目计划
- 通知相关方
- 跟踪实施情况

**变更控制委员会（CCB）**
定期（如每周）召开变更评审会，集中处理变更请求，避免频繁打断开发。

**减少变更的策略**：
1. 前期充分的需求调研和验证
2. 迭代开发，早期交付可用版本
3. 设置需求冻结期（如上线前2周）
